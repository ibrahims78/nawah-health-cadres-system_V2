
ضافة "تصدير قالب مشروع موجود" ماذا تعني به وما المعلومات المصدرة وكيف يتم اعادة استيرادها وهل سيصب هناك تعارض في المشاريع بالمنصة او ستم اضافة مشروع جديد واعدادات جديدة اخبرني بالتفاصيل دون ان تعدل شئ

تصدير قالب مشروع موجود — الشرح الكامل
ماذا يعني؟
ببساطة: تأخذ مشروعاً قائماً بكل إعداداته وحقوله وتُحوّله إلى ملف YAML يمكن رفعه لاحقاً لإنشاء مشروع جديد تماماً بنفس البنية.

ما الذي يُصدَّر؟
# كل ما يُصدَّر من المشروع الحالي:
project:
  name: "نموذج تسجيل الموظفين"
  description: "..."
  form:
    title: "سجّل بياناتك"
    subtitle: "..."
    enabled: true
    editTokenHours: 48
    invitationCode: "STAFF-ABC123"
  steps:
    - "البيانات الشخصية"
    - "المستندات"
    - "المراجعة"
fields:
  - key: "full_name"
    label: "الاسم الكامل"
    type: text
    required: true
    step: 1
  - key: "gender"
    label: "الجنس"
    type: radio
    options: ["ذكر", "أنثى"]
    step: 1
  - key: "cv_file"
    label: "السيرة الذاتية"
    type: file
    fileSettings:
      allowedTypes: ["pdf", "docx"]
      maxSizeMb: 5
    step: 2
  # ... باقي الحقول كاملة
integrations:
  google_sheets:
    enabled: true
    spreadsheet_id: "1BxiMVs0XRA5nFMdKvBdBZjgmUUqptlbs74OgVE2upms"
    sheet_name: "بيانات"
  telegram:
    enabled: false   # ← التوكن لا يُصدَّر أبداً لأسباب أمنية
    chat_id: "-1001234567890"

ما الذي لا يُصدَّر أبداً؟
البيانات	السبب
سجلات المستخدمين (الردود/البيانات)	هذه بيانات وليست إعدادات
telegramBotToken	بيانات اعتماد حساسة مشفّرة
googleServiceAccountKey	بيانات اعتماد حساسة مشفّرة
driveOAuthClientSecret + refresh token	بيانات اعتماد حساسة
الملفات المرفوعة في uploads/	هذه بيانات مستخدمين
سجل النشاط (audit log)	تاريخي لا قيمة له كقالب
معرّف المشروع (id)	يتولّد جديداً عند الاستيراد
كيف يعمل الاستيراد؟
المدير يرفع ملف YAML
         ↓
الخادم يُحلّل الملف ويتحقق منه (Zod validation)
         ↓
يعرض تقرير معاينة قبل الإنشاء:
  "سيُنشأ مشروع جديد باسم: نموذج تسجيل الموظفين"
  "عدد الحقول: 12 حقلاً"
  "تحذير: Telegram chat ID موجود لكن bot token غير مضمّن — أضفه من الإعدادات"
         ↓
المدير يضغط "إنشاء"
         ↓
POST /api/projects/  ← مشروع جديد بـ ID جديد + الحقول
PATCH /api/projects/:newId ← الإعدادات الآمنة (sheet ID، chat ID)
         ↓
النتيجة: مشروع جديد مستقل تماماً

هل سيكون هناك تعارض مع المشاريع الموجودة؟
لا — أبداً.

السبب الجوهري: الاستيراد يُنشئ مشروعاً جديداً دائماً بـ UUID جديد. لا يُعدّل أي مشروع موجود، ولا يتشارك معه أي بيانات. المشاريع في المنصة مستقلة تماماً عن بعضها.

قبل الاستيراد:          بعد الاستيراد:
─────────────           ─────────────────────────────
مشروع A (id: aaa)  →   مشروع A (id: aaa)  ← لم يتغيّر
مشروع B (id: bbb)  →   مشروع B (id: bbb)  ← لم يتغيّر
                    →   مشروع C (id: ccc)  ← جديد من القالب

التعارض الوحيد المحتمل نظرياً هو اسم المشروع — لكن المنصة لا تمنع مشروعين بنفس الاسم (الـ ID هو المعرّف الفعلي، وليس الاسم).

القيمة العملية للميزة
السيناريو الأبرز: مؤسسة لديها قالب موحّد (مثلاً "نموذج طلب إجازة") تريد نشره في 5 فروع مختلفة، كل فرع له حسابه الخاص على المنصة. المدير يُصدّر القالب مرة واحدة ويُرسله للفروع الخمسة، كل فرع يرفعه وينشئ مشروعه المستقل بنفس البنية خلال ثوانٍ.


هل يمكن ان يتضمن تصدير المشروع بيانات اعتماد حساسة مشفّرة لتصبح نسخة احتياطية من المشروع مع اعداداته او اضافة مشروع جديد مع الاعدادت وبعد التصدير يمكن تعديل اي اعداد اخبرني بالامكانية دون ان تعدل شئ

تضمين بيانات الاعتماد الحساسة في تصدير المشروع — التحليل الكامل
الإمكانية
نعم، ممكن تقنياً — لكنه يتحول من "قالب" إلى "نسخة احتياطية مشروع" وهو تمييز مهم. المنصة لديها البنية اللازمة لكليهما.

التحدي الجوهري: التشفير عبر البيئات
البيانات الحساسة مشفّرة بـ ENCRYPTION_KEY الخاص بالبيئة الحالية:

ENCRYPTION_KEY(بيئة A) → encrypt("ya29.token...") → "xKj9mP..."

عند التصدير، هناك ثلاثة خيارات لمعالجة هذه البيانات:

الخيار الأول: تصدير كـ "قالب نظيف" (بدون اعتمادات)
integrations:
  telegram:
    bot_token: ""      # ← فارغ، يملؤه المستخدم بعد الاستيراد
    chat_id: "-100123"

الاستخدام: نشر القالب لأشخاص آخرين، إنشاء مشاريع مشابهة. القيد: يحتاج إعادة إدخال كل الاعتمادات يدوياً.

الخيار الثاني: تصدير مشفّر بكلمة مرور المصدِّر
# مشابه تماماً لما صمّمناه في backup-restore-feature.md
_export_meta:
  encryption: "password-protected"
  kdf: "pbkdf2"
  salt: "a3f9bc..."
  
integrations:
  telegram:
    bot_token_encrypted: "Bx7qK..."   # مشفّر بكلمة مرور المُصدِّر
    chat_id: "-100123"
  google_sheets:
    service_account_key_encrypted: "Ym3nR..."

عند التصدير:

1. فك تشفير الحقل بـ ENCRYPTION_KEY البيئة الحالية
   decrypt("xKj9mP...") → "ya29.token..."
2. أعد تشفيره بكلمة مرور يُدخلها المدير
   PBKDF2(password, salt) → backupKey
   encrypt("ya29.token...", backupKey) → "Bx7qK..."
3. ضعه في ملف التصدير

عند الاستيراد:

1. المدير يُدخل كلمة المرور
2. فك تشفير بـ backupKey
3. أعد تشفير بـ ENCRYPTION_KEY البيئة الجديدة
4. أنشئ المشروع بالاعتمادات كاملة

الاستخدام: نسخة احتياطية من المشروع، نقل المشروع بين حسابات.

الخيار الثالث: تصدير مشفّر بمفتاح البيئة (نفس الخادم فقط)
_export_meta:
  encryption: "environment-key"
  # تعمل فقط عند الاستيراد على نفس الخادم
  
integrations:
  telegram:
    bot_token_enc: "xKj9mP..."   # مشفّر كما هو في DB

الاستخدام: نسخة احتياطية سريعة ضمن نفس المنصة. القيد: لا تعمل على خادم آخر بمفتاح مختلف.

مقارنة الخيارات الثلاثة
قالب نظيف	مشفّر بكلمة مرور	مشفّر بمفتاح البيئة
قابل للنقل بين خوادم	✅	✅	❌
قابل للمشاركة مع آخرين	✅	⚠️ (بكلمة المرور)	❌
يحتفظ بالاعتمادات	❌	✅	✅
يحتاج كلمة مرور	❌	✅	❌
أمان الملف المُصدَّر	عالٍ	عالٍ	متوسط
الاستخدام الأمثل	قوالب قابلة للنشر	نسخ احتياطية كاملة	نسخ سريعة داخلية
هل الاستيراد ينشئ مشروعاً جديداً دائماً؟
دائماً — بغض النظر عن الخيار المُختار. حتى لو كانت الملف يحتوي على كل بيانات الاعتماد:

قبل الاستيراد:                بعد الاستيراد:
─────────────                 ─────────────────────────────
مشروع A (id: aaa) → موجود    مشروع A (id: aaa) → لم يتغيّر
                               مشروع B (id: fff) → جديد ← من الملف

هذا يُميّزه جوهرياً عن ميزة النسخ الاحتياطي والاستعادة التي تستعيد المشروع على نفس الـ ID وتكتب فوق البيانات القديمة.

هل يمكن تعديل الإعدادات بعد الاستيراد؟
نعم، بشكل كامل. المشروع المُنشأ مشروع عادي تماماً — الـ PATCH /api/projects/:id يعمل عليه كأي مشروع آخر. يمكن للمدير بعد الاستيراد مباشرة:

تعديل أي حقل أو إضافة حقول جديدة
تغيير Telegram bot token بآخر
ربط Google Sheets مختلف
تفعيل/تعطيل النموذج
تعيين كود دعوة جديد
الفارق بين هذه الميزة وميزة backup-restore
تصدير/استيراد مشروع	نسخ احتياطي واستعادة
وحدة العملية	مشروع واحد	كامل المنصة
النتيجة	مشروع جديد دائماً	استعادة كل شيء كما كان
يشمل بيانات المستخدمين	❌	✅
يشمل الملفات المرفوعة	❌	✅
هدفه	نسخ البنية والإعدادات	التعافي من الكوارث
يُلغي ما هو موجود	❌ أبداً	✅ إذا اختار المدير ذلك
الخلاصة
الميزة الأمثل عملياً هي تصدير بـ نمطين يختارهما المدير لحظة التصدير:

"قالب قابل للمشاركة" — بدون اعتمادات، لنشر القالب لأشخاص آخرين
"نسخة كاملة للمشروع" — بكلمة مرور، تحفظ كل الاعتمادات وتُستخدم للاسترداد أو النقل بين بيئات
وفي كلتا الحالتين: الاستيراد = مشروع جديد، لا تعارض، وكل الإعدادات قابلة للتعديل بعده.


انشئ ملف احترافي وتفصيلي للخلاصة الميزة الأمثل عملياً هي تصدير بـ نمطين يختارهما المدير لحظة التصدير:

"قالب قابل للمشاركة" — بدون اعتمادات، لنشر القالب لأشخاص آخرين "نسخة كاملة للمشروع" — بكلمة مرور، تحفظ كل الاعتمادات وتُستخدم للاسترداد أو النقل بين بيئات وفي كلتا الحالتين: الاستيراد = مشروع جديد، لا تعارض، وكل الإعدادات قابلة للتعديل بعده. واجعل الملف جاهز لتنفيذ الميزة بالكامل عند الحاجة واتاكد انه ظاهر في المشروع
